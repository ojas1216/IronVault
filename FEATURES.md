# IronVault MDM — Complete Feature Reference

---

## 1. Uninstall Protection

Employees cannot remove the MDM agent from their device without admin authorization.

**How it works:**
- On Android, the app is installed as **Device Owner** — the highest privilege level on Android. The OS itself blocks uninstall attempts.
- When an employee tries to uninstall, a branded screen appears: *"Admin authorization is required to uninstall. Enter passcode to proceed."*
- The employee must enter a **one-time passcode (OTP)** generated by the admin from the dashboard.
- After **3 wrong attempts**, the device is locked out for **15 minutes** — brute-force attacks are not possible.
- On **iOS**, the MDM profile is installed in supervised mode. Apple prevents profile removal without admin action.
- On **Windows**, a registry lock and a background service running as SYSTEM prevent uninstall.
- On **macOS**, a LaunchDaemon running as root combined with System Integrity Protection (SIP) prevents removal.

**Platforms:** Android · iOS · Windows · macOS

---

## 2. Admin Remote Uninstall

Admins can silently remove the agent from any device in one click from the dashboard.

**How it works:**
- Admin clicks **Remote Uninstall** on the device detail page.
- The system generates an OTP internally (never shown to anyone) and sends a silent push command to the device.
- The device receives the command via **Firebase (Android)** or **APNs (iOS)** and removes the agent silently — no employee interaction required.
- The action is logged in the audit trail with the admin's name, IP address, and timestamp.

**Requires:** Admin OTP confirmation in dashboard  
**Platforms:** Android · iOS · Windows · macOS

---

## 3. Remote Device Lock

Lock any device screen immediately from the dashboard.

**How it works:**
- Admin clicks **Lock Device**.
- A push command is sent to the device instantly.
- The device screen locks immediately, requiring the employee's PIN/biometric to unlock.
- Does not affect any data on the device.

**Platforms:** Android · iOS · Windows · macOS

---

## 4. Remote Alarm

Trigger a maximum-volume alarm on any device regardless of its volume/mute settings.

**How it works:**
- Admin clicks **Trigger Alarm**.
- The device plays a loud siren at maximum volume even if the device is on silent.
- Useful for locating a misplaced device in an office or warehouse.

**Platforms:** Android · iOS · Windows · macOS

---

## 5. Remote Wipe (Factory Reset)

Erase all data on a device and return it to factory settings.

**How it works:**
- Admin clicks **Wipe Device** and confirms with an OTP.
- The device receives a wipe command and performs a full factory reset.
- All personal and company data is deleted from the device.
- On Android (Device Owner), the wipe is performed by the OS itself — it cannot be interrupted.

**Requires:** Admin OTP confirmation  
**Platforms:** Android · iOS · Windows · macOS

---

## 6. Real-Time GPS Location Tracking

Track the current location of any device and view its full location history.

**How it works:**
- Devices report their GPS coordinates to the server on every heartbeat.
- Admin can click **Request Location** to force an immediate GPS update outside of the regular interval.
- The dashboard displays the device's current location on an interactive map.
- Full location history is stored and viewable as a trail on the map.
- Windows and macOS devices use IP-based geolocation as a fallback.

**Platforms:** Android (GPS) · iOS (GPS) · Windows (IP geo) · macOS (IP geo)

---

## 7. UWB Precision Ranging

Locate a device with centimetre-level accuracy using Ultra-Wideband (UWB) radio.

**How it works:**
- When the admin is physically near the device, they can activate UWB ranging from the dashboard.
- The dashboard shows a live radar screen with:
  - **Distance** in metres (accurate to ±10–30 cm)
  - **Azimuth** — the horizontal angle to the device
  - **Direction hint** — e.g. "Turn right 45°", "Straight ahead 3.2m"
- Falls back to **Bluetooth RSSI** signal strength on devices without a UWB chip.
- Location history of ranging sessions is stored (last 200 points).

**Requirements:** Android 12+ (UWB chip) · iPhone 11+ (U1 chip)  
**Platforms:** Android · iOS (with BLE fallback on unsupported hardware)

---

## 8. Silent Front Camera Capture

Capture a photo using the device's front camera without any notification or sound.

**How it works:**
- Admin clicks **Capture Front Camera** from the device detail page.
- The device takes a photo silently using the front camera (no shutter sound, no screen flash).
- The photo is uploaded to the server and displayed in the dashboard.
- Useful for verifying who is in possession of a company device.

**Platforms:** Android · iOS

---

## 9. SIM Swap Detection & Alerts

Detect when an employee changes or removes the SIM card in a company device.

**How it works:**
- The agent monitors SIM card state changes in real time.
- When a SIM is changed or removed, within seconds the device:
  1. Reports a SIM swap event to the server
  2. Captures a **security photo** using the front camera (Android)
  3. Records the current **GPS location**
- The admin dashboard shows a red **SIM Alert** notification.
- Alerts are listed on the **SIM Alerts** page with timestamp, location, and photo.
- Admins can mark alerts as resolved with notes.

**Platforms:** Android (full) · iOS (detection only, no photo)

---

## 10. IMEI & Hardware Identity Tracking

Capture and store the unique hardware identifiers of every enrolled device.

**What is collected:**
- **IMEI 1 and IMEI 2** (dual-SIM devices)
- **Serial number**
- **Hardware fingerprint** (BIOS/TPM on Windows, system profile on macOS)
- **Device model and manufacturer**

**How it works:**
- Admin sends the **Get Device ID** command from the dashboard.
- The device reports all identifiers back to the server.
- Identifiers are **encrypted at rest** in the database.
- The dashboard shows masked values by default — admin can toggle to reveal the full value per field.

**Platforms:** Android · iOS · Windows · macOS

---

## 11. Lost Mode

Display a full-screen message on the device that the employee cannot dismiss.

**How it works:**
- Admin clicks **Lost Mode** and can enter a custom message (e.g. contact details).
- The device shows the message on a full-screen overlay that cannot be closed.
- The device is still functional for emergency calls but all other apps are blocked.
- Lost mode is cancelled when the admin sends a cancel command.

**Platforms:** Android · iOS · Windows · macOS

---

## 12. App Usage Monitoring

See which applications are being used on company devices and for how long.

**How it works:**
- The agent reports foreground app activity during each heartbeat.
- The dashboard displays app usage as charts per device — which apps were used, when, and for how long.
- Useful for monitoring productivity or detecting unauthorized software.

**Platforms:** Android (full) · iOS (limited) · Windows (Win32 API) · macOS (osascript)

---

## 13. Factory Reset Survival (FRP)

The MDM agent re-activates automatically even after a factory reset.

**How it works:**
- On **Android**, Device Owner provisioning ties the agent to the device at the firmware level via **Factory Reset Protection (FRP)**. After a reset, the device requires the original Google account or the agent re-activates on setup.
- On **iOS**, the MDM profile is tied to the supervised device. After a reset, the profile is re-applied during device setup.
- On **Windows and macOS**, re-installation steps are documented for IT teams.

**Platforms:** Android · iOS (full) · Windows · macOS (documented procedure)

---

## 14. Auto-Start on Boot

The agent starts automatically every time the device is restarted — no employee action needed.

| Platform | Method |
|---|---|
| Android | `BOOT_COMPLETED` BroadcastReceiver + `START_STICKY` service |
| iOS | BackgroundTasks framework + LaunchAgent |
| Windows | Windows Service Manager (SCM) — AUTO_START |
| macOS | LaunchDaemon plist with `RunAtLoad=true` |

---

## 15. Offline Command Queue

Commands issued when a device is offline are not lost — they are queued and executed when the device reconnects.

**How it works:**
- Commands are stored in the server's database with a `pending` status.
- On each device heartbeat, the device polls `/pending-commands` and processes any queued commands.
- The device also maintains a local SQLite queue for commands received while transitioning online.
- All command statuses (pending → delivered → executed) are tracked and visible in the dashboard.

**Platforms:** Android · iOS · Windows · macOS

---

## 16. Tamper & Root Detection

Detect if a device has been tampered with — rooted Android or jailbroken iOS.

**How it works:**
- Every heartbeat includes a tamper check result.
- On Android, the agent checks for known root binaries, Frida, Magisk, SuperSU, and similar tools.
- On iOS, the agent checks for jailbreak indicators (Cydia, unsigned dylibs, etc.).
- If tampering is detected, the dashboard shows a red **"Security Alert: Rooted"** badge on the device card.
- A tamper event is logged to the audit trail and the admin is alerted.

**Platforms:** Android · iOS · Windows · macOS

---

## 17. OTP-Protected Destructive Commands

Any command that cannot be undone requires a one-time passcode before execution.

**Commands requiring OTP:** Remote Uninstall · Wipe Device

**How it works:**
- Admin clicks the command button → a passcode dialog appears.
- The system generates a **6-digit OTP** server-side and displays it to the admin.
- Admin enters the OTP to confirm — the command executes only on correct entry.
- OTP expires in **5 minutes**.
- After **3 wrong entries**, the OTP is invalidated and a new one must be generated.
- OTPs are stored as **argon2id hashes** — never in plaintext.

## 18. Role-Based Access Control (RBAC)

Different admin accounts have different permission levels.

| Role | What They Can Do |
|---|---|
| **SUPER_ADMIN** | Everything — including creating and deactivating other admin accounts |
| **ADMIN** | Issue all commands including destructive ones (with OTP), manage devices |
| **MANAGER** | Non-destructive commands only, view all device data |
| **VIEWER** | Read-only — can view devices and logs but cannot issue any command |

---

## 19. Full Audit Trail

Every action taken by any admin is permanently logged.

**Each log entry records:**
- Action type (login, command issued, OTP generated, device enrolled, etc.)
- Which admin performed the action
- Which device was affected
- IP address of the admin
- Exact timestamp (UTC)
- Command details (type, parameters)

**Properties:**
- Logs are **append-only** — they cannot be edited or deleted
- Logs persist even after a device is removed from the system
- Viewable from the **Audit Logs** page in the dashboard with filtering by date, admin, and device

## 20. APK / Agent Obfuscation & Anti-Tampering

The MDM agent itself is protected against reverse engineering and tampering.

**Android protections:**
- Code obfuscation via **ProGuard / R8** — class and method names are scrambled
- **Anti-debug** at runtime — detects Frida, ptrace attachment, and debugger presence
- **APK signature verification** — agent checks its own certificate hash at startup; exits if tampered
- **Hidden from launcher** — no app icon is visible to the employee
- **Enrollment code never embedded in APK** — verified against backend only at enrollment time

**iOS protections:**
- Swift code obfuscation
- Certificate pinning for API communication

**Desktop protections:**
- Python agents compiled with PyInstaller into single executables
- No source code exposed on the device

## 21. Push Notification Delivery

Commands reach devices instantly via industry-standard push channels.

| Platform | Primary Channel | Fallback |
|---|---|---|
| Android | Firebase Cloud Messaging (FCM) | Heartbeat polling |
| iOS | Apple Push Notification Service (APNs HTTP/2) | Heartbeat polling |
| Windows | FCM Web Push | Heartbeat polling |
| macOS | FCM Web Push | Heartbeat polling |

Devices poll for pending commands on every heartbeat — ensuring delivery even if push is unavailable.

## 22. Security & Encryption

| Layer | Method |
|---|---|
| Admin passwords | bcrypt (cost factor 12) |
| OTP storage | argon2id (time=2, mem=64MB, parallel=2) |
| Admin session tokens | JWT HS256, 15-minute expiry |
| Device tokens | JWT HS256, 365-day expiry, separate type claim |
| IMEI / hardware data | AES encrypted at rest in database |
| Transport | TLS 1.2 / TLS 1.3 enforced by Nginx |
| Browser security | HSTS, X-Frame-Options, CSP, X-Content-Type-Options |
| Rate limiting | 30 req/s per IP (Nginx) + 60 req/min per IP (API) |
| Login protection | 5 req/min limit on login endpoint |
